# Q&A: Reweight 调整触发数据迁移的机制与源码实现

## 问题
在 1.3.3 节中提到通过调整 reweight 可以触发 PG 在 OSD 之间进行迁移。
**疑问**：
1. 这个迁移仅仅是映射关系的变化，还是实际数据的物理迁移？
2. 调整 reweight 后，似乎新数据会被分配到不同的地方，那“触发迁移”具体是如何发生的？是写新数据时才发生，还是旧数据会被主动搬迁？
3. 关于 reweight 的生效和数据迁移，代码实现主要在 Ceph 的哪一部分？

## 解答
### 1. 迁移的性质：实际数据拷贝
1.3.3 节提到的迁移，**绝不仅仅是逻辑上的映射变化，而是实实在在的“物理数据迁移”**。
这意味着数据会通过网络从旧的 OSD 传输到新的 OSD，并写入磁盘。在 Ceph 术语中，这通常被称为 **Backfill（回填）** 或 **Recovery（恢复）**。

### 2. 触发机制与触发点
你提到“影响新数据存储”是完全正确的，但 Ceph 的核心设计不仅是让新数据去往正确的地方，还要**保证旧数据也回到正确的位置**。

*   **逻辑变更（映射变化）**：
    当你修改 `reweight` 时，Monitor 会更新全局的 CRUSH Map（OSDMap）。
*   **触发过程**：
    1.  **计算新家**：集群中所有的 OSD 节点收到新的 Map 后，会重新计算自己负责的所有 PG 的归属（即计算 `up` 集合）。
    2.  **发现错位**：由于权重变了，CRUSH 算法会算出：*“原本属于 OSD A 的某些 PG，现在应该搬到 OSD B 去了。”*
    3.  **触发迁移**：此时，即便没有任何**新数据写入**，OSD A 也会发现自己手里的数据变成了“多余”或“错位”的。
*   **触发点**：
    触发点是 **OSD 进程加载并识别到了 OSDMap 的变更**。一旦 OSD 发现当前的数据持有状态（Current Set）与最新的 Map 计算结果（Up Set）不一致，它就会自动触发后台数据迁移机制，将旧数据搬运给 OSD B。

### 3. 源码实现位置
这个过程贯穿了 Ceph 的三个核心组件：

#### A. 指令入口（Monitor）
用户输入 `ceph osd reweight <id> <weight>`
*   **文件**：`src/mon/OSDMonitor.cc`
*   **行为**：Monitor 接收命令，修改内存中的配置（如 `new_weight`），然后生成新的 OSDMap 版本并发布给全集群。

#### B. 状态检测（OSD）
OSD 收到新 Map 后，决定是否需要迁移。
*   **文件**：`src/osd/PG.cc` (或 `UpmapItem` 相关逻辑)
*   **关键逻辑**：
    *   OSD 调用 CRUSH 算法重新计算 `new_up`（新的 OSD 列表）。
    *   将其与 `current_up`（当前持有数据的 OSD 列表）进行比对。
    *   如果发现不一致（例如 OSD B 出现在新列表中但不在旧列表中），PG 的状态会变为 `Peering` 或 `Backfilling`。

#### C. 数据搬运（OSD）
执行实际的拷贝操作。
*   **文件**：`src/osd/PrimaryLogPG.cc` (或旧版的 `ReplicatedPG.cc`)
*   **关键逻辑**：
    *   通过 `handle_peering_event` 处理状态机流转。
    *   调用 `do_recovery` 或 `start_primary_lease` 等函数。
    *   主 OSD 将对象数据打包，通过网络发送给新 OSD（目标节点），目标节点将其写入 BlueStore/RocksDB。

### 总结
`reweight` 的改变就像更改了“户籍管理处”的归属规则。
1.  **新出生的人（新数据）**：按新规则直接落户新家。
2.  **原有的人（旧数据）**：发现由于规则改变，自己的户口归属地变了。户籍管理处（OSD）会立刻启动“搬迁程序”，把人实际搬（拷贝）到新家。
