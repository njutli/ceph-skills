# 第3章纠删码在Ceph中的生效原理

## 核心概念：PG与分片 (Shard)
在纠删码（EC）存储池中，逻辑 PG 的物理映射与多副本不同：
- **映射关系**：1个逻辑 PG 转换为 $K+M$ 个唯一的 PG 实例（即分片，Shard）。
- **身份标识**：每个 Shard 有唯一的 Shard ID ($0 \dots K+M-1$)。
- **Primary Shard**：
  - 从 $K+M$ 个 OSD 中选出第一个作为 Primary Shard（由 CRUSH 规则决定）。
  - 它是所有读写请求的**协调者**，负责切片、编码、分发以及读取聚合。
  - 其他 OSD（Secondary Shard）只是单纯的存储单元，不知道数据的全貌。

## 读写操作模型

### 1. 新写 (Whole Stripe Write)
- 客户端发请求给 Primary Shard。
- Primary 将对象切分为 **Stripe**（条带），每个 Stripe 包含 $K$ 个数据块。
- 计算 $M$ 个校验块。
- 将 $K+M$ 个 Chunk 并行分发给 $K+M$ 个 OSD 写入。
- **特点**：这是最理想的写场景，不需要读取旧数据。

### 2. 覆盖写 (Overwrite & RMW)
- **挑战**：修改部分数据会导致校验块失效，必须重新计算。
- **RMW 流程 (Read-Modify-Write)**：
  1. **补齐读**：如果写入没有覆盖完整条带（Partial Write），Primary 必须先从其他 OSD 把该条带剩余的数据块读回来。
  2. **重算**：凑齐旧数据块和新数据块，重新生成新的校验块。
  3. **写回**：将新数据块和新校验块写回 OSD。
- **优化**：
  - **写缓冲 (Write Cache)**：Ceph 尝试合并小写请求，凑成 Whole Stripe 再写，避免 RMW。
  - **部分补齐读**：只读取覆盖写缺失的那部分块，而不是读取整个条带。
  > **代价**：RMW 会导致严重的读放大和延迟增加，这是 EC 不适合随机小写的主要原因。

### 3. 读操作
- **异步读取**：Primary 向各 OSD 异步发读请求，拼装后返回客户端。
- **条带对齐**：读请求通常需要对齐到 Stripe 边界，因为解码是以 Chunk 为单位的（尤其在降级模式下需要用校验块修复数据时）。

## 术语速查
- **Chunk (块)**：编码产生的最小单元。
- **Stripe (条带)**：一次编码产生的所有 Chunk 集合（$K$ 个数据块 + $M$ 个校验块）。
- **Shard (分片)**：同一对象属于同一个 Shard ID 的所有 Chunk 集合（即一个 OSD 上的数据）。
