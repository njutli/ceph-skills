# BlueStore 写回调 on_commit 与 on_readable 执行顺序机制解析

## 1. 核心疑问与书中矛盾
书中 2.6.4 节提到：
1.  **定义**：`on_readable` 代表写日志完成；`on_commit` 代表数据写入主要存储设备完成。
2.  **前提**：“BlueStore 写日志要先于写数据完成”。
3.  **结论**：“BlueStore 会先执行 on_commit，然后才执行 on_readable”。

**矛盾点**：如果日志先完成，理应是 `on_readable` 先触发。书中结论与前提直接冲突。

## 2. 真实机制解析：数据先于日志
书中的前提描述存在**笔误**。在 BlueStore 的真实实现中，物理写入顺序其实是 **数据先于日志**：

### 2.1 为什么数据必须先写？
BlueStore 遵循严格的 **WAL（预写日志）语义**，这与文件系统（如 Ext4 的 Journal 模式）保证一致性的原理如出一辙：
*   **数据（Data）**：通过 `libaio` 直接写入裸盘（Main Device）。
*   **日志（Metadata/WAL）**：写入 RocksDB（kvDB），记录 Onode 和 Extent 映射。
*   **一致性约束**：RocksDB 的提交（日志落盘）**必须等待数据块 IO 完成**。如果日志先落盘记录了数据位置，但数据还没写进去，此时掉电，重启后根据日志恢复就会读到脏数据。
*   **类比**：就像 Ext4 的 Journal 模式，必须先确保数据块落盘，才能通过日志将元数据落盘。否则，元数据指向了一个无效的数据块，文件系统的一致性就被破坏了。

### 2.2 回调触发顺序
基于上述物理顺序：
1.  **数据写入完成**：触发 `on_commit`（书中定义的数据落盘信号）。
2.  **日志提交完成**：触发 `on_readable`（书中定义的日志落盘信号）。

因此，**`on_commit` 必然先于 `on_readable` 执行**。

## 3. 总结
书中“写日志要先于写数据完成”的描述是错误的。
正确的逻辑是：**BlueStore 中数据直接写裸盘，且日志（元数据）提交必须等待数据 IO 完成。因此，数据落盘（on_commit）先于日志落盘（on_readable）。**
