# Q&A: BlueStore 覆盖写机制详解

在 2.6.4 节中，BlueStore 的覆盖写（overwrite）处理逻辑较为复杂，涉及 MAS 切分、COW/RMW/WAL 三种策略的组合。本节通过具体例子详细拆解。

## 前置概念

假设磁盘块大小 = 4KB，MAS（最小可分配空间）= 4KB。

BlueStore 的写策略总原则：
- **MAS 对齐的中间部分** → **COW**（写时重定向，分配新块）
- **非 MAS 对齐的头尾部分** → **RMW**（读-改-写，原地更新）

---

## 举例：向 offset 5000 写入 12KB 数据

假设对象中 offset 5000 之前已有数据（所以这是**覆盖写**）。

### 第一步：按 MAS 边界切分

```
写入范围：offset 5000, length 12KB (12288 bytes)

MAS边界（4KB对齐）:
  0      4096     8192     12288    16384    20480
  |------|--------|--------|--------|--------|
       ^5000                          ^17088
       |<------12KB写入范围---------->|
       
头部非对齐: [5000, 8192)     → 3192 bytes（不足一个MAS）
中间MAS对齐: [8192, 12288)   → 4096 bytes（1个完整MAS）
             [12288, 16384)  → 4096 bytes（1个完整MAS）
尾部非对齐: [16384, 17088)   → 704 bytes（不足一个MAS）
```

### 第二步：中间 MAS 对齐部分 → COW

**[8192, 12288)** 和 **[12288, 16384)** 这两段是 MAS 对齐的。

BlueStore 对这两段执行 **COW**：在磁盘上新分配空间，把新数据写进去，然后修改 Onode 的 ExtentMap 指针，指向新块。旧块标记为"待释放"，等事务组完成后归还 Allocator。

> 覆盖写和新写在 MAS 对齐部分的处理一样，唯一区别是：覆盖写需要找出旧 Extent 并标记释放，新写不存在旧 Extent。

### 第三步：头尾非对齐部分 → 三个分支判断

以**头部 [5000, 8192)** 为例（3192 bytes，不足一个 4KB 块）：

#### 分支 1：能否跳过 RMW，直接 COW？

如果这个 Extent 被压缩过，要做 RMW 就得先读→解压→改→重压，代价太大。此时直接 COW：分配新块，新数据写入，旧 Extent 释放。

> 但 COW 有代价：新块中 offset [0, 5000) 和 [8192, 4096) 是"空穴"（属于旧数据但不被新写覆盖），后续读这块需要读两次再 Merge，性能下降。

#### 分支 2：能否复用已有 Extent 的 unused 块？

BlueStore 的 Extent 内部以**基本块**（比如 4KB）为单位管理。如果 MAS > 基本块大小，Extent 中可能有 unused 块。unused 块里没有有效数据，对它写入不怕掉电写坏。

但本例中 MAS = 基本块大小 = 4KB，没有 unused 块可以复用，所以跳过。

#### 分支 3：WAL 写（最兜底、最慢）

当前两种都不行时，RMW 不可避免。但 RMW 有掉电写坏风险，所以需要 **WAL（Write-Ahead Log）** 保护。

流程（对应书上图 2-16）：

```
1. 补齐读：把旧数据中 [4096, 8192) 这整个块从磁盘读上来
2. 合并：   把新写的 3192 bytes 合并到该块的 [5000-4096, 8192-4096) 位置
3. 填充：   块头部 [0, 5000-4096) 用全 0 填充（或保留旧数据）
4. 生成 WAL 日志事务，写入 kvDB（RocksDB/WAL）
5. 等 WAL 写入 kvDB 成功后 → 才能原地覆盖写回磁盘 [4096, 8192)
6. 覆盖写完成后 → 删除 WAL 日志事务条目
```

尾部 [16384, 17088) 处理逻辑类似。

### 第四步：TransContext 三阶段执行

所有修改操作加入 TransContext 后，处理分三阶段：

| 阶段 | 内容 | 掉电影响 |
|------|------|----------|
| (1) 等待 COW 的写 I/O 完成 | 中间对齐部分的 COW 写入新地址 | 掉电无影响（kvDB 还没更新指针，旧数据还在） |
| (2) 同步元数据到 kvDB | 新空间分配、旧空间释放、ExtentMap 更新等 | ACID 保证：要么全完成，要么无影响 |
| (3) 通过 WAL 执行覆盖写 | 头尾非对齐部分的原地覆盖写 | WAL 已在 kvDB 中，掉电后可重放恢复 |

---

## 总结

覆盖写的核心逻辑：

1. **按 MAS 切分**：中间对齐部分走 COW（和新写一样，多一步释放旧块）；头尾非对齐部分走三条分支
2. **头尾非对齐部分优先级**：直接 COW → 复用 unused 块 → WAL 写
3. **WAL 写最慢**：补齐读 + 合并 + 填充 → 写 WAL 到 kvDB → 原地覆盖写 → 删除 WAL
4. **执行顺序**：COW 写 I/O（阶段1）→ kvDB 同步（阶段2）→ WAL 覆盖写（阶段3），保证任何阶段掉电都不丢数据
