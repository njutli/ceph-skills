# 第4章表4-1核心概念解析

本笔记用于记录表4-1中对 Ceph 核心术语的讨论与辨析。

---

## 1. Up Set 与 Acting Set

### 1.1. 定义辨析
| 概念 | 通俗理解 | 决策依据 |
|:---|:---|:---|
| **Up Set** | "户口本"（理论名单 / 计划） | **CRUSH 算法**。规定了在当前拓扑规则下，这些分片**应该**存储在哪些 OSD 上（即数据的最终归宿）。 |
| **Acting Set** | "打工人"（实际名单 / 现实） | **当前实际状况**。规定了此时此刻**实际在干活并能响应客户端 IO** 的 OSD。 |

### 1.2. 排序规则
这两个集合中的顺序**不按 OSD ID 数字大小排序**，而是遵循：
1.  **第一位（Index 0）**：必须是 **Primary OSD**（读写入口，协调者）。
2.  **后续位置**：由 CRUSH 算法针对该 PG 计算出的哈希伪随机顺序决定 Secondary OSDs。
这种有序性确保了全网对 "谁负责哪个分片 ID" 达成共识。

### 1.3. 经典场景：数据迁移中的状态不一致
通常发生在**扩容、缩容或 CRUSH 规则调整**导致的数据搬迁（Remapping）期间。

**示例：计划迁移**
假设一个 3 副本存储池：
1.  **正常状态**：CRUSH 算出的理想名单 `[1, 2, 3]`。
    *   `Up = [1, 2, 3]`, `Acting = [1, 2, 3]`（一致）。
2.  **触发变更**：管理员调整规则，决定将 PG 数据从 OSD 1 移到 OSD 4。CRUSH 算出的新目标变成了 `[4, 2, 3]`。
3.  **Remapped 状态**：此时数据还在 1 上，4 刚加入是空的。
    *   **Up Set**：立即更新为 `[4, 2, 3]`（这是目标）。
    *   **Acting Set**：保持为 `[1, 2, 3]`（这是现实）。

**注意**：此处的 "代班"（Acting Set 仍含 1）前提是 **OSD 1 没有宕机**。如果硬件真坏了，它无法处理 IO，Acting Set 中会直接将其剔除。

### 1.4. 为什么必须有 Up Set？
仅保留 `Acting Set` 无法实现 "不停机搬家"，`Up Set` 的存在解决了以下核心问题：
- **提供导航目的地**：迁移过程依赖 `Up Set` 知道要把数据从哪里同步给谁（比如从 Acting 的 1 同步给 Up 的 4）。
- **触发 `remapped` 状态**：当 `Up != Acting` 时，PG 进入 `remapped` 标记。Primary 据此执行前端正常响应的业务逻辑，同时后台将 IO 流量复制给目标 OSD。
- **应对并发变更**：如果在迁移中又改了规则（如目标变成 5），`Up Set` 会立即更新为最新拓扑状态，确保迁移方向永远指向最新的 CRUSH 结果，不会陷入死锁。
- **决定恢复策略**：故障恢复时，若某 OSD 仍在 `Up Set` 中，系统会判定它是合法编制，需等待其恢复或从其他副本重建；若已不在 `Up Set` 中，说明已被 CRUSH 淘汰，可直接加速新节点顶替。

## 2. Peering 与 Authoritative History

### 2.1. 核心定义
- **Peering（协同/对等协商）**：
  当 PG 成员发生变化（如掉线、重启、新节点加入）后，所有幸存的 PG 实例暂停服务，聚在一起“重新对账”的过程。目的是选出新的主节点（Primary）、对齐数据状态，最终恢复对外服务。
- **Authoritative History（权威日志）**：
  Peering 过程中确立的“真理”。它是指在所有幸存的 PG 实例中，**版本最新、最完整的那份 Log**。其他落后或数据丢失的实例必须以此为准进行同步恢复。

### 2.2. 核心场景示例

假设一个 **3 副本** 的 PG 1.0，运行在 OSD 1（Primary）、OSD 2、OSD 3 上。

1.  **初始正常状态**
    客户端陆续写入数据，版本号为 `v1:1`, `v1:2`, `v1:3`。Log 同步到所有副本：
    *   OSD 1/2/3 的 Log 尾部（last_update）均为 `v1:3`。
    *   数据状态完全一致。

2.  **故障发生**
    **OSD 1 掉线**。集群 Epoch 增加，CRUSH 重新计算，OSD 4 顶替 OSD 1。
    *   **Up Set / Acting Set**：`[4, 2, 3]`。
    *   **OSD 4**：是空壳，没有任何数据。
    *   **OSD 4** 被选为新的 Peering Primary，它启动 Peering 过程。

3.  **Peering 第一步：交换 Info，选出权威日志**
    OSD 4 向所有同伴索要 Info（含 last_update 等元数据）：
    
    | OSD | Peer 汇报状态 | 备注 |
    |:---|:---|:---|
    | **OSD 4** | 无（空） | 新顶替的 Primary，啥也没有 |
    | **OSD 2** | last_update: `v1:3` | 存活，持有旧数据 |
    | **OSD 3** | last_update: `v1:3` | 存活，持有旧数据 |
    
    **OSD 4 的判定规则**：
    *   **OSD 2 和 3 的 last_update 相同且最大**。
    *   **结论**：**OSD 2 或 3 的 Log 就是 Authoritative History（权威日志）**。

    > **关键点**：这里的“选举”并不是 Paxos/Raft 那种全网投票，而是基于**版本号（Eversion）大小的逻辑比对**。谁的版本最大，谁就是真理。

4.  **Peering 第二步：基于权威日志进行数据同步**
    OSD 4 确定权威日志后，开始自我恢复：
    *   **拉取**：从 OSD 2 拉取完整 Log `[v1:1, v1:2, v1:3]`。
    *   **比对**：发现自己是空的，缺失从 v1:1 开始的所有数据。
    *   **执行同步（Backfill）**：从权威 OSD（OSD 2）全量拷贝对象。
    
    同步完成后，OSD 4 的数据补齐，PG 1.0 恢复 Active 状态。

### 2.3. 变体：不完全掉线（增量同步）
如果不是 OSD 1 掉线导致空 OSD 顶替，而是**OSD 3 短暂断网**，错过了一次写入。
*   **OSD 1**：持有 `v1:4`（最新）。
*   **OSD 3**：断网恢复前停留在 `v1:3`。
*   **Peering 判定**：OSD 1 的 Log 为 Authoritative History。OSD 3 只需增量拉取 v1:4。

## 3. Interval（区间）

### 3.1. 什么是 Interval？
简单来说，**Interval 是 Up Set 和 Acting Set 保持恒定的那段时间（即 Epoch 的序列）。**

只要集群的拓扑结构发生变化（比如 OSD 加入、掉线、权重调整），导致 **Up Set**（理论名单）或 **Acting Set**（实际名单）任意一个发生变化，旧的 Interval 就会结束，新的 Interval 随即开启。
每个 Interval 都有一个唯一的起始 Epoch 编号（`same_interval_since`）。

### 3.2. 结合案例理解 Interval
回到笔记中"OSD 4 顶替 OSD 1"的例子，我们可以看到 Interval 是如何划分的：

*   **Interval 1 (过去)**：
    *   **Epoch 范围**：`100` -> `105`
    *   **状态**：在这 6 个 Epoch 期间，一切风平浪静。
    *   **映射**：`Up = [1, 2, 3]`, `Acting = [1, 2, 3]`。
    *   **日志**：客户端的所有写入（如 `v100:1` 到 `v100:5`）都发生在这个 Interval 内，此时大家公认 1, 2, 3 负责处理数据。

*   **事件触发**：
    *   **Epoch 106**：管理员调整了规则（或者 OSD 1 确认宕机），Mon 发布了新的 OSDMap。
    *   CRUSH 重新计算得出新的 **Up Set = `[4, 2, 3]`**。

*   **Interval 2 (现在)**：
    *   **Epoch 范围**：从 `106` 开始...
    *   **变化**：因为 Up Set 变了，**Interval 1 结束，Interval 2 开启**。
    *   **状态**：
        *   此时 `Up = [4, 2, 3]`（新目标）。
        *   此时 `Acting = [2, 3]`（暂时降级，因为 4 还没数据）或者 `Acting = [4, 2, 3]`（如果迁移正在进行中，4 开始参与工作）。
    *   **关键点**：只要进入 Interval 2，所有新的写入请求（如 `v100:6`）都必须由新的一组 Acting OSD 来处理。

### 3.3. Interval 的核心意义
为什么 Peering 机制要如此关注 Interval？

1.  **判定数据的合法性（防冲突）**：
    如果 OSD 2 在 Interval 1 期间记录了某次写入，而 OSD 4 在 Interval 2 也记录了写入。Peering 时会发现它们属于**不同的 Interval**，从而根据 Epoch 的大小判断谁的数据是“后来的”，谁的是“旧的”，避免数据混乱。
2.  **Peering 的“起跑线”**：
    当新的 Primary 上位（OSD 4），它进行 Peering 的目的之一，就是确认**当前 Interval** 开始的那个时间点，大家的状态是否一致。
    如果发现有 OSD 落后了（比如还停留在 Interval 1 的状态），就知道它缺席了整个 Interval 2 期间的更新，需要进行全量或增量同步。

### 3.4. 关键误区：数据迁移完成不会结束当前 Interval

**结论：** 迁移完成（即 Acting Set 追上 Up Set）时，**不会**开启新的 Interval。

- **Interval 的边界**：由 **OSDMap Epoch 的变化** 决定。只有当 OSDMap 发生变化（如 OSD 宕机、权重改变、CRUSH 规则修改），旧 Interval 才会结束。
- **迁移完成只是内部动作**：当数据从 `[1, 2, 3]` 搬运到 `[4, 2, 3]` 完毕时，这仅仅是 PG 内部状态的一次翻转（从 `remapped` 恢复为 `active`）。这个过程发生在 Interval 2 **内部**，并没有触发 Mon 发布新的 OSDMap。
- **何时变成 Interval 3？** 只有当迁移完成后，外部再次发生事件（例如 **OSD 2 坏了**，或者 **管理员又改了规则**），迫使 Mon 发布新的 OSDMap，此时 Interval 2 才会结束，Interval 3 才会开始。

## 4. PGBackend（副本/纠删码策略引擎）

### 4.1. 核心定义
**PGBackend** 是 PG 内部的一个插件化模块，负责**“将针对原始对象的操作转化为副本之间的分布式操作”**。

简单来说，PG 负责处理"业务逻辑"（如：检查权限、分配版本号、管理状态），但它**不关心**数据到底是怎么存的（是存 3 份，还是切片计算校验码）。
这个“怎么存”的脏活累活，全交给 **PGBackend**。它主要有两种实现：
*   **ReplicatedBackend**：对应多副本。简单粗暴，直接复制数据。
*   **ECBackend**：对应纠删码。负责切片、计算校验码、分发分片。

---

### 4.2. 场景示例：一条写入请求的“分流”

假设客户端向 PG 1.0 发送了一个写入请求：**写入对象 `Object A`（内容 4MB）**。
PG 主节点接收到请求，处理完权限检查和分配版本号后，会将这个任务交给 `PGBackend` 执行。此时发生了有趣的分流：

#### 场景 A：存储池类型是“多副本（3副本）”
此时 PG 绑定的是 **ReplicatedBackend**。

1.  **PG 的动作**：告诉 Backend，“把 `Object A` 写进去”。
2.  **ReplicatedBackend 的反应**：*“收到，3 副本策略。”*
3.  **执行过程**：
    *   它在本地写完一份完整的 `Object A`。
    *   同时，它将 `Object A` 的完整副本发送给 **Secondary OSDs**（比如 OSD 2 和 3）。
    *   等待 OSD 2 和 3 回复"写入成功”。
    *   向 PG 汇报：“搞定”。

#### 场景 B：存储池类型是“纠删码（4+2）”
此时 PG 绑定的是 **ECBackend**。

1.  **PG 的动作**：同样告诉 Backend，“把 `Object A` 写进去”。（PG 甚至不知道底层是 EC）。
2.  **ECBackend 的反应**：*“收到，执行纠删码策略。”*
3.  **执行过程**：
    *   **切片**：将 4MB 数据切成 4 个 1MB 的数据块 `D1, D2, D3, D4`。
    *   **计算**：利用公式计算出 2 个校验块 `P1, P2`。
    *   **分发**：将这 6 个块（`D1~D4, P1~P2`）分别发送给对应的 6 个分片 OSD。
    *   等待所有 OSD 回复"写入成功"。
    *   向 PG 汇报：“搞定”。

### 4.3. 总结
对于上层的 PG 逻辑来说，场景 A 和场景 B **毫无区别**——它只负责确保数据的版本和状态正确。
**PGBackend** 屏蔽了底层存储策略的复杂性，使得 Ceph 可以灵活切换副本或纠删码模式，而无需重写 PG 的核心代码。

### 4.4. 实体形态：代码模块而非进程
需要明确的是，PGBackend **不是**一个独立运行的进程（Process）。

它是 `ceph-osd` 进程内部的一个类组件（Class Instance），与 PG 实例一一对应。
```text
OSD 进程 (内存空间)
 ├── PG 1.0 (属于副本池)
 │    └── [PGBackend] 绑定 ReplicatedBackend 实例
 ├── PG 2.1 (属于 EC 池)
 │    └── [PGBackend] 绑定 ECBackend 实例
 └── ...
```
同一个 `ceph-osd` 进程可以同时承载多个不同类型的 PG。当 PG 对象被创建时，系统根据它所属的存储池类型，动态加载对应的 PGBackend 实现。

## 5. PG Temp (临时映射重定向)

### 5.1. 核心定义
PG Temp 是 Monitor 或管理员下发的一个**临时指令**，用于**强制修改特定 PG 的 Up Set 和 Acting Set 的优先级顺序**。

它主要解决一个死锁问题：当 CRUSH 算出来的“理论名单”（Up Set）无法完成 **Peering（对账恢复）**时（例如排在第一位的 Primary OSD 宕机，导致剩下的 OSD 不敢出头当主），通过 PG Temp 强制指定一个**存活的、有数据的 OSD** 来临时充当 Primary，带头完成 Peering。

### 5.2. 场景示例：Primary 宕机后的僵局与解救

假设一个 3 副本存储池，PG 1 的正常映射由 CRUSH 算法决定：
- **默认顺序**：`[OSD 1, OSD 2, OSD 3]`
- **角色**：**OSD 1** 排在第一位，它是法定的 **Primary OSD**。

#### 1. 故障与僵局
- **OSD 1 突然宕机 (Down)**。
- OSDMap 更新，但 CRUSH 算法基于目前的拓扑，计算出的 Up Set **依然是** `[OSD 1, OSD 2, OSD 3]`（因为规则没变，理论上 1 还是老大）。
- 此时，OSD 2 和 OSD 3 依然存活，它们想要尝试进行 **Peering** 来恢复 PG 1。
- **问题出现**：
  根据规则，只有排在第一位的才是 Primary。OSD 2 和 OSD 3 发现自己不是“理论上的老大”，它们不敢擅自做主，会一直尝试联系 OSD 1。由于 OSD 1 已经挂了，Peering 就会卡死（PG 状态变为 `peering` 或 `down`），整个 PG **无法对外提供服务**。

#### 2. PG Temp 介入 (破局)
为了打破僵局（通常由 Monitor 自动触发或由管理员手动执行命令），集群下发一条 PG Temp 指令：
> **“对于 PG 1，临时把名单顺序改为：[OSD 2, OSD 1, OSD 3]！”**

- OSD 2 和 OSD 3 收到指令后，发现 OSD 2 排在了第一位。
- **OSD 2** 此时立即**越权上位**，成为临时 Primary。

#### 3. 完成 Peering 恢复服务
- **OSD 2 (临时 Primary)** 开始主持 Peering 大局。它检查自己和 OSD 3 谁的 Log 更新，确定权威日志并完成同步。
- PG 1 成功切换为 `active` 状态，恢复读写服务。

#### 4. PG Temp 的消除
- 一段时间后，坏掉的 OSD 1 修好了并重新上线。
- Monitor 确认 PG 数据已完整，OSD 1 具备了重新当老大的能力。
- Monitor **撤销** `pg_temp` 指令。
- 顺序变回默认的 `[OSD 1, OSD 2, OSD 3]`，OSD 1 恢复 Primary 身份。

### 5.3. 总结
PG Temp 是 Ceph 的一种**“变通”机制**。当死板的 CRUSH 计算结果（Up Set）因为硬件故障导致恢复流程受阻时，PG Temp 通过修改顺序让“活下来的人”带头干活，从而保障集群的可用性。

## 6. 基础术语关联 (Log, Info, Eversion)

这三个术语是理解 Peering 和同步机制的基础：

1.  **Eversion (Epoch + Version)**：
    版本号的精确表示（如 `v1:3`，即 Epoch 为 1，Version 为 3）。它是判断谁拥有“权威日志”的唯一度量标准。
    
2.  **Log**：
    记录了所有 Eversion 的操作序列。Peering 本质上是各个 OSD 交换 Log，比对 Eversion。
    
3.  **Info**：
    Log 的摘要，仅包含了 PG 的基本元数据和当前最高 Eversion。为了减少通信开销，Peering 的第一步是各节点互相交换 Info 而不是完整的 Log，以此来快速找到拥有权威日志的节点。
