# 第4章表4-1核心概念解析

本笔记用于记录表4-1中对 Ceph 核心术语的讨论与辨析。

---

## 1. Up Set 与 Acting Set

### 1.1. 定义辨析
| 概念 | 通俗理解 | 决策依据 |
|:---|:---|:---|
| **Up Set** | "户口本"（理论名单 / 计划） | **CRUSH 算法**。规定了在当前拓扑规则下，这些分片**应该**存储在哪些 OSD 上（即数据的最终归宿）。 |
| **Acting Set** | "打工人"（实际名单 / 现实） | **当前实际状况**。规定了此时此刻**实际在干活并能响应客户端 IO** 的 OSD。 |

### 1.2. 排序规则
这两个集合中的顺序**不按 OSD ID 数字大小排序**，而是遵循：
1.  **第一位（Index 0）**：必须是 **Primary OSD**（读写入口，协调者）。
2.  **后续位置**：由 CRUSH 算法针对该 PG 计算出的哈希伪随机顺序决定 Secondary OSDs。
这种有序性确保了全网对 "谁负责哪个分片 ID" 达成共识。

### 1.3. 经典场景：数据迁移中的状态不一致
通常发生在**扩容、缩容或 CRUSH 规则调整**导致的数据搬迁（Remapping）期间。

**示例：计划迁移**
假设一个 3 副本存储池：
1.  **正常状态**：CRUSH 算出的理想名单 `[1, 2, 3]`。
    *   `Up = [1, 2, 3]`, `Acting = [1, 2, 3]`（一致）。
2.  **触发变更**：管理员调整规则，决定将 PG 数据从 OSD 1 移到 OSD 4。CRUSH 算出的新目标变成了 `[4, 2, 3]`。
3.  **Remapped 状态**：此时数据还在 1 上，4 刚加入是空的。
    *   **Up Set**：立即更新为 `[4, 2, 3]`（这是目标）。
    *   **Acting Set**：保持为 `[1, 2, 3]`（这是现实）。

**注意**：此处的 "代班"（Acting Set 仍含 1）前提是 **OSD 1 没有宕机**。如果硬件真坏了，它无法处理 IO，Acting Set 中会直接将其剔除。

### 1.4. 为什么必须有 Up Set？
仅保留 `Acting Set` 无法实现 "不停机搬家"，`Up Set` 的存在解决了以下核心问题：
- **提供导航目的地**：迁移过程依赖 `Up Set` 知道要把数据从哪里同步给谁（比如从 Acting 的 1 同步给 Up 的 4）。
- **触发 `remapped` 状态**：当 `Up != Acting` 时，PG 进入 `remapped` 标记。Primary 据此执行前端正常响应的业务逻辑，同时后台将 IO 流量复制给目标 OSD。
- **应对并发变更**：如果在迁移中又改了规则（如目标变成 5），`Up Set` 会立即更新为最新拓扑状态，确保迁移方向永远指向最新的 CRUSH 结果，不会陷入死锁。
- **决定恢复策略**：故障恢复时，若某 OSD 仍在 `Up Set` 中，系统会判定它是合法编制，需等待其恢复或从其他副本重建；若已不在 `Up Set` 中，说明已被 CRUSH 淘汰，可直接加速新节点顶替。

## 2. Peering 与 Authoritative History

### 2.1. 核心定义
- **Peering（协同/对等协商）**：
  当 PG 成员发生变化（如掉线、重启、新节点加入）后，所有幸存的 PG 实例暂停服务，聚在一起“重新对账”的过程。目的是选出新的主节点（Primary）、对齐数据状态，最终恢复对外服务。
- **Authoritative History（权威日志）**：
  Peering 过程中确立的“真理”。它是指在所有幸存的 PG 实例中，**版本最新、最完整的那份 Log**。其他落后或数据丢失的实例必须以此为准进行同步恢复。

### 2.2. 核心场景示例

假设一个 **3 副本** 的 PG 1.0，运行在 OSD 1（Primary）、OSD 2、OSD 3 上。

1.  **初始正常状态**
    客户端陆续写入数据，版本号为 `v1:1`, `v1:2`, `v1:3`。Log 同步到所有副本：
    *   OSD 1/2/3 的 Log 尾部（last_update）均为 `v1:3`。
    *   数据状态完全一致。

2.  **故障发生**
    **OSD 1 掉线**。集群 Epoch 增加，CRUSH 重新计算，OSD 4 顶替 OSD 1。
    *   **Up Set / Acting Set**：`[4, 2, 3]`。
    *   **OSD 4**：是空壳，没有任何数据。
    *   **OSD 4** 被选为新的 Peering Primary，它启动 Peering 过程。

3.  **Peering 第一步：交换 Info，选出权威日志**
    OSD 4 向所有同伴索要 Info（含 last_update 等元数据）：
    
    | OSD | Peer 汇报状态 | 备注 |
    |:---|:---|:---|
    | **OSD 4** | 无（空） | 新顶替的 Primary，啥也没有 |
    | **OSD 2** | last_update: `v1:3` | 存活，持有旧数据 |
    | **OSD 3** | last_update: `v1:3` | 存活，持有旧数据 |
    
    **OSD 4 的判定规则**：
    *   **OSD 2 和 3 的 last_update 相同且最大**。
    *   **结论**：**OSD 2 或 3 的 Log 就是 Authoritative History（权威日志）**。

    > **关键点**：这里的“选举”并不是 Paxos/Raft 那种全网投票，而是基于**版本号（Eversion）大小的逻辑比对**。谁的版本最大，谁就是真理。

4.  **Peering 第二步：基于权威日志进行数据同步**
    OSD 4 确定权威日志后，开始自我恢复：
    *   **拉取**：从 OSD 2 拉取完整 Log `[v1:1, v1:2, v1:3]`。
    *   **比对**：发现自己是空的，缺失从 v1:1 开始的所有数据。
    *   **执行同步（Backfill）**：从权威 OSD（OSD 2）全量拷贝对象。
    
    同步完成后，OSD 4 的数据补齐，PG 1.0 恢复 Active 状态。

### 2.3. 变体：不完全掉线（增量同步）
如果不是 OSD 1 掉线导致空 OSD 顶替，而是**OSD 3 短暂断网**，错过了一次写入。
*   **OSD 1**：持有 `v1:4`（最新）。
*   **OSD 3**：断网恢复前停留在 `v1:3`。
*   **Peering 判定**：OSD 1 的 Log 为 Authoritative History。OSD 3 只需增量拉取 v1:4。

## 3. 基础术语关联 (Log, Info, Eversion)

这三个术语是理解 Peering 和同步机制的基础：

1.  **Eversion (Epoch + Version)**：
    版本号的精确表示（如 `v1:3`，即 Epoch 为 1，Version 为 3）。它是判断谁拥有“权威日志”的唯一度量标准。
    
2.  **Log**：
    记录了所有 Eversion 的操作序列。Peering 本质上是各个 OSD 交换 Log，比对 Eversion。
    
3.  **Info**：
    Log 的摘要，仅包含了 PG 的基本元数据和当前最高 Eversion。为了减少通信开销，Peering 的第一步是各节点互相交换 Info 而不是完整的 Log，以此来快速找到拥有权威日志的节点。
