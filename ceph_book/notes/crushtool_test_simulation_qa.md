# Q&A: `crushtool` 模拟测试参数与结果解析

## 问题
在阅读 1.3.1 节“（5）模拟测试”时，书中给出了两个 `crushtool` 命令的例子。
**疑问**：这两个例子中，哪些是输入？哪些是输出？各个参数和结果的具体含义是什么？为什么第二个例子的范围是 100000？

## 解答
`crushtool` 模拟测试命令用于在**不实际部署集群**的情况下，验证 CRUSH 规则和拓扑配置的合理性与均衡性。

### 示例一：查看具体映射列表（微观验证）
**命令**：
`crushtool -i mycrushmap --test --min-x 0 --max-x 9 --num-rep 3 --ruleset 0 --showMappings`

#### 输入参数解析
* **`-i mycrushmap`**: 输入的 CRUSH Map 文件。
* **`--test`**: 标记进入测试模式，不对集群产生副作用。
* **`--min-x 0 --max-x 9`**: **输入值范围**。`x` 代表 PG ID 或 Object ID。这里模拟了 **10 个 PG** (从 0 到 9)。
* **`--num-rep 3`**: 指定副本数量为 3（即要求每个 PG 选出 3 个 OSD）。
* **`--ruleset 0`**: 指定使用的规则编号为 0。
* **`--showMappings`**: **输出模式**。要求列出每个 `x` 具体落在哪些设备上。

#### 输出结果解析
```text
CRUSH rule 0 x 0 [19,11,3]
CRUSH rule 0 x 1 [15,7,21]
...
```
* **格式**: `CRUSH rule [规则ID] x [输入ID] [OSD1, OSD2, OSD3]`
* **含义**:
    * `x 0 [19,11,3]`: PG 0 的数据副本被分配给了 OSD 19、11 和 3。
* **用途**: **检查逻辑正确性**。人工检查输出的列表，确认选出的 OSD 是否符合规则要求（例如：3 个 OSD 是否真的位于不同的 Host 上，是否没有冲突）。

### 示例二：查看宏观统计分布（宏观均衡）
**命令**：
`crushtool -i mycrushmap --test --min-x 0 --max-x 100000 --num-rep 3 --ruleset 0 --show/utilization`

#### 输入参数解析
* **`--max-x 100000`**: **关键变化**。模拟范围扩大到了 **10 万**。这是为了获得**统计学上的准确性**。样本越小，随机性干扰越大；样本越大，越能反映真实的整体数据分布情况。
* **`--show/utilization`**: **输出模式**。要求输出统计摘要，而不是罗列 10 万条记录。

#### 输出结果解析
```text
rule 0 (replicated/ruleset), x = 0..100000, numrep = 3..3
rule 0 (replicated/ruleset) num_rep 3 result size == 3: 100001/100001
device 0: stored : 11243 expected : 11111.2
device 1: stored : 11064 expected : 11111.2
...
```
* **第一/二行**: **概要信息**。
    * `result size == 3: 100001/100001`: 表示所有 100,001 个模拟输入都**成功**分配到了 3 个 OSD。如果有失败的情况（比如某 Host 磁盘不足），这里的数值会小于分母。
* **后续行**: **统计信息**。
    * `device [ID]`: 具体的 OSD 编号。
    * `expected`: **理论期望值**。计算公式约为：`(PG总数 × 副本数) / OSD总数`。这里 `100001 * 3 / 27 ≈ 11111.2` (假设有 27 个 OSD)。假设所有磁盘完全一样，理论上每个盘该存这么多。
    * `stored`: **实际计算值**。CRUSH 算法实际分配给该 OSD 的副本数量。
* **用途**: **检查均衡性**。
    * 对比 `stored` 和 `expected`，如果两者非常接近，说明算法的**哈希分布非常均匀**。
    * 如果某个盘的 `stored` 远超 `expected`，说明该盘是“热点”，可能存在 CRUSH 规则配置不当或层级冲突问题。
