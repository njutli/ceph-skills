# 2.5.2 BlueFS 日志机制与日志压缩解析

## 问题：BlueFS 的日志为什么“只增不减”？与传统日志有何不同？

### 解答
传统认知中的“日志”通常指预写式日志（WAL），仅作为数据损坏时的保险副本；而 BlueFS 采用的是**日志结构文件系统（Log-Structured File System）**设计，两者的底层逻辑有本质区别。

---

## 1. 核心差异：传统日志 vs. 日志结构

| 特性 | 传统日志 (ext4/XFS WAL) | BlueFS (日志结构) |
|:---|:---|:---|
| **日志的角色** | 保险副本。存在独立的主元数据结构（如 inode 表、位图）。 | **唯一真相源**。无独立静态元数据表，日志本身就是持久化状态。 |
| **写入方式** | 随机写为主，日志仅用于保证操作原子性。 | 纯顺序追加写。所有操作（创建/删除/改属性）均追加至单一日志文件。 |
| **释放时机** | 数据/元数据落盘并同步后，立即截断释放日志空间。 | **永不主动释放**，直到执行日志压缩（Compaction）。 |
| **上电恢复** | 仅重放未提交完成的日志条目，恢复后丢弃。 | 必须重放完整日志链（或最新快照+增量）以重建内存视图。 |

---

## 2. 为什么需要“遍历所有日志”重放？
因为 BlueFS **没有**传统的目录树或 inode 表持久化在磁盘上。内存中的 `dir_map` 和 `file_map` 是管理核心，但掉电即失。
磁盘上唯一保留的，就是那条不断增长的日志文件。因此：
* **上电时 (Log Replay)**：BlueFS 只能从头到尾读取日志，将每一条操作在内存中“重新执行”一遍，最终拼凑出完整的文件系统视图。

---

## 3. 日志压缩的本质与举例
日志只增不减会导致空间耗尽且上电重放时间指数级增长。**日志压缩（Compaction）**的作用就是丢弃无效历史，只保留最新有效状态。

### 实例推演
假设执行以下操作序列：
1. 创建文件 `file`
2. 写入数据 `AAA`
3. 删除文件 `file`
4. 再次创建文件 `file`
5. 写入数据 `BBB`

* **压缩前**：磁盘上存在 5 条顺序日志。上电需全部重放。
* **压缩后**：BlueFS 会将当前内存中的最终状态（文件 `file` 存在，内容为 `BBB` 的元数据记录）序列化为**一条基准快照日志**追加到末尾，并直接截断删除该快照之前的所有 5 条历史日志。
* **效果**：逻辑上等效于“创建文件 `file` -> 写入 `BBB`”，但实际实现为**单次全量状态快照**。下次上电只需加载该快照，瞬间还原全貌，彻底避免了长链路重放。

### 总结
你的疑惑源于将 BlueFS 的“日志”等同于传统文件系统的 WAL。实际上，BlueFS 的日志是**存储元数据的主体**。

---

## 4. 完整存储流程实例：目录与文件的落盘

BlueFS 中的“文件、目录、日志”与常规文件系统有显著差异。以**RocksDB 请求创建目录 `/cf_1` 并向文件 `data.sst` 写入数据 "AAAA"**为例，磁盘上的真实存储过程如下：

### 第一阶段：创建目录 `/cf_1`
1. **内存操作**：BlueFS 在内存的 `dir_map` 中记录 `/cf_1` -> DirID=2。
2. **落盘日志**：追加日志条目到 Log File 尾部：
   *   `[Txn 1] OP_DIR_CREATE: name="/cf_1", id=2`
   *   **注意**：此时磁盘上并没有独立的“目录”文件实体，只有这行记录。

### 第二阶段：写入文件 `data.sst` (内容: "AAAA")
1. **分配空间**：BlueFS 在数据盘分配一段物理块（假设起始地址 `0x2000`）。
2. **写入数据**：将 "AAAA" **直接写入物理块 `0x2000`**。这是纯粹的物理数据存储。
3. **更新内存**：`file_map` 更新文件 Inode 信息（大小 4KB，指向 Extent `0x2000`）。
4. **落盘日志**：追加日志条目到 Log File 尾部，告知系统文件 `data.sst` 的数据在哪：
   *   `[Txn 2] OP_FILE_UPDATE: ino=10, extents=[(0x2000, 4K)]`

### 总结：磁盘上到底有什么？
当我们去查看磁盘时，会看到两种截然不同的东西：
*   **Log File (日志)**：这是一个文件，包含了从格式化以来的**所有元数据变更**（创建、删除、修改）。它是元数据的唯一持久化位置，没有传统的 inode 表或目录树。
*   **Data Blocks (数据文件)**：这是散落分布的物理空间，保存着真实的 RocksDB 数据（如 `data.sst` 的内容）。

**上电恢复 (Replay) 的原理**：
由于磁盘上没有静态的目录结构，BlueFS 启动时必须读取 Log File，从 `[Txn 1]` 开始重放：
*   读到 `OP_DIR_CREATE` -> 在内存建立目录。
*   读到 `OP_FILE_UPDATE` -> 在内存建立文件节点，并指向物理块。
这样，内存中就重建出了“目录树”和“文件表”，让系统恢复工作。
