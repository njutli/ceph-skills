# Q&A: 数据映射与物理拷贝的关系

## 问题
书中提到 "straw 算法会随机地将一些原有元素中的数据重新映射至新加入的元素之中"。
**疑问**：
1. "数据映射" 到底是怎样一个过程？
2. 映射过程直接涉及到物理数据的拷贝吗？

## 解答
**结论**：映射（Mapping）本身只是**算出来的逻辑关系**，不直接涉及数据拷贝，但映射关系确定后，会**触发**底层数据的物理拷贝（称为 Backfill 或重均衡）。

这个过程可以拆解为两步：

1. **找新家（逻辑映射 - CRUSH 计算）**
   * **过程**：当新 OSD 加入时，客户端或主 OSD 使用 CRUSH 算法重新计算：*“对于现有的海量数据块（Object），哪一部分现在的归属权属于这个新 OSD？”*
   * **特点**：纯数学计算，速度极快，只改变“谁归谁管”的元数据记录，**不涉及实际的数据传输**。

2. **搬行李（物理拷贝 - Backfill/Rebalance）**
   * **过程**：一旦逻辑映射计算出“某块数据应该归新 OSD 管”，系统会发现这块数据其实还在老 OSD 上。
   * **动作**：主 OSD（Primary）会在后台将这块数据的物理副本**复制（Copy）**发送给新的 OSD。
   * **术语**：这个物理搬运的过程在 Ceph 中叫做 **Backfill（回填）** 或 **Rebalance（重均衡）**。

### 为什么强调“随机”映射？
* **笨办法的后果**：如果使用简单的求余算法（如 `Hash % N`），加入新节点后，几乎所有数据的位置都会变化，导致 **99% 的数据都要被重新拷贝**，集群会瞬间瘫痪。
* **CRUSH/Straw 的优势**：它的设计目标是让**只有极少部分**“命中”新计算逻辑的数据发生位置改变。
* **效果**：这就像你搬新家，只有客厅的几件家具（被随机选中的数据）需要搬运，卧室里的东西（绝大部分旧数据）还在原地不动。

### 总结
* **映射** = **决定**谁该搬（大脑思考）。
* **拷贝** = **执行**搬运动作（手脚行动）。
