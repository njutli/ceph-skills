# Ceph PG 分裂（PG Splitting）详解：原理、推演与实践

Ceph 的 PG 分裂（PG Splitting）是当集群扩容或数据增多时，增加一个存储池中 Placement Group (PG) 数量的过程。

为了理解分裂，我们先通过一个比喻来建立直观认识，再用一个详细的例子拆解整个过程。

### 🧠 理解 PG 分裂：从“书架”到“书箱”

Ceph 存储数据的过程引入了 PG 这个中间层，可以把它想象成搬家时的**书箱**：

1.  **第一层映射：对象 → PG (书 → 书箱)**
    Ceph 先将所有数据（书）通过哈希算法均匀地分配到各个 PG（书箱）中。某个对象属于且只属于一个 PG。

2.  **第二层映射：PG → OSD (书箱 → 房间)**
    然后，Ceph 的 CRUSH 算法再将这些 PG（书箱）分配到集群的各个 OSD（存储房间）中。

所以，一个 OSD 里会存放很多个 PG（很多书箱）。PG 分裂的实质是：**增加书箱的总数，并把旧书箱里的书重新整理到新书箱中去。**

### ⚙️ 核心概念解析：`pg_num` 与 `pgp_num`

在深入例子前，必须先理清两个容易混淆的关键参数：

*   **`pg_num`**: PG 的数量。你可以理解为**书箱的个数**。增加 `pg_num`，就是在规划中增加了书箱的总数，这是分裂的**第一步**。
*   **`pgp_num`**: 用于 PG 放置的 PG 数量。你可以理解成**已经“贴好标签、摆上货架”的书箱个数**。`pgp_num` 决定了有多少个书箱真正参与到 CRUSH 算法的计算和分配中。

**关键规则**：
*   **`pgp_num` 不能超过 `pg_num`**。
*   **分裂分两步走**：先增加 `pg_num`（宣布有更多箱子了），再增加 `pgp_num`（决定哪些箱子放在哪儿）。当 `pg_num > pgp_num` 时，Ceph 会报告健康警告（HEALTH_WARN）。

### 📚 详细举例：从 4 个 PG 分裂到 8 个 PG

假设我们有一个存储池叫 `mypool`，目前有 4 个 PG，每个 PG 里都有一些对象。

**初始状态 (`pg_num = 4`, `pgp_num = 4`)**
PG 的分布和对象归属如下表所示：

| 原 PG ID | 包含的对象（简化） | 存放的 OSD（副本位置） |
| :--- | :--- | :--- |
| PG 1 | Object A, E, I... | osd.1, osd.2, osd.3 |
| PG 2 | Object B, F, J... | osd.1, osd.2, osd.4 |
| PG 3 | Object C, G, K... | osd.2, osd.3, osd.4 |
| PG 4 | Object D, H, L... | osd.1, osd.3, osd.4 |

**对象归属计算方式（底层实际使用位掩码优化版伪代码）**：
`pg_id = hash(object_name) & (pgp_num - 1)`  
*(注：逻辑上等价于取模，但底层使用位运算加速，这也是必须倍增的原因)*

#### 第一阶段：增加 `pg_num`

我们决定将 PG 总数从 4 增加到**8**。管理员执行命令：`ceph osd pool set mypool pg_num 8`。

1.  **计算新归属**：Ceph Monitor 更新了 OSD Map。现在新对象的存储计算方式掩码变长。
2.  **规划分裂方案**：每个旧 PG 将分裂成两个新 PG。分裂方案由哈希算法决定，通常是 `新 PG_id_1 = 旧 PG_id`，`新 PG_id_2 = 旧 PG_id + 旧 pg_num`。所以：
    *   PG 1 将分裂为 **PG 1** 和 **PG 5**
    *   PG 2 将分裂为 **PG 2** 和 **PG 6**
    *   PG 3 将分裂为 **PG 3** 和 **PG 7**
    *   PG 4 将分裂为 **PG 4** 和 **PG 8**
3.  **本地“逻辑分裂”**：**这个阶段基本没有数据迁移！** 每个 OSD 只是在本地为那些被规划出的新 PG 创建了新的目录（子 PG）。原本存储在 PG 1 的对象 A、E、I 等，还待在原地（PG 1 的目录下），只是它们的“身份”被标记为“尚未归位”。

#### 第二阶段：增加 `pgp_num`

现在 `pg_num=8`，但 `pgp_num=4`。Ceph 知道有了新箱子，但还没决定新箱子放在哪个房间。这时，执行命令 `ceph osd pool set mypool pgp_num 8`，触发真正的数据迁移。

CRUSH 算法会为新的 PG 5-8 计算出新的、完全独立的主 OSD 位置。例如：
*   **PG 5** 可能被计算出应放置在 **(osd.2, osd.3, osd.1)**
*   **PG 1** 仍保持在 **(osd.1, osd.2, osd.3)**

现在，Ceph 开始数据“归位”，将对象移动到正确的 PG 目录中：
*   **对象 A**：原本在 PG 1。根据新算法，可能发现它仍然应该属于 PG 1。那么它**不需要移动**。
*   **对象 E**：原本也在 PG 1。但根据新算法，它可能被计算出属于**PG 5**。那么 Ceph 就会将对象 E 的副本从 PG 1 的目录**迁移**到 PG 5 的目录，并发送到 PG 5 的主 OSD（osd.2）上。

这个过程中，只有那些“归属”发生变化的数据（约 50%）才会在集群内真正迁移，而未变化的则保留在原地。

### ⏱️ 影响与优化

PG 分裂是一个**后台高负载操作**，因为它涉及数据迁移，会大量消耗磁盘 I/O 和网络带宽。

为了减少对业务的影响，Ceph 提供了调整迁移速度的参数。例如，可以限制后台恢复和回填操作的并发度：

```bash
# 限制每个 OSD 同时进行的恢复操作数
ceph tell osd.* injectargs '--osd-recovery-max-active 1'
# 限制每个 OSD 同时进行的回填操作数
ceph tell osd.* injectargs '--osd-max-backfills 1'
```
*(注：Ceph 的限流机制主要基于**并发线程数**和**I/O 优先级**控制，而非按数据总量的固定百分比切分。默认 `osd-max-backfills=1` 确保了分裂是排队进行的，避免一拥而上打满带宽。)*

此外，Ceph 的**PG 自动扩展（PG Auto-Scaling）**功能可以帮你避免手动操作的麻烦。它会根据集群的容量和使用情况，自动建议或调整 `pg_num` 的值。

### ⚠️ 实践准则

尽管逻辑上 PG 分裂是“本地”的，但随后的数据重分布会产生实际的集群负载。你需要注意：

1.  **逐步增加**：手动调整时，建议 `pg_num` 每次翻倍（如 4->8->16），而不是一次性调得过大，以匹配底层位运算的平滑切分特性。
2.  **监控集群**：在分裂过程中，密切监控集群的 `recovering` 和 `misplaced` 状态。Ceph 默认会通过并发控制限制迁移负载，以减少性能冲击。

PG 分裂是 Ceph 集群扩容和维护中的常规操作。理解 `pg_num` 和 `pgp_num` 的微妙区别，是掌握 Ceph 数据分布逻辑的关键一步。

## 💡 延伸探讨：关于“OSD 扩容”与“PG 必须倍增”的深度答疑

**问题**：
书中提到“如果 PG 数目不能被 2 整除”的情况。如果不倍增，按书中解释也是可以正常映射的，那 PG 数量增加时不倍增会有什么问题吗？倍增我理解是个很大的变动，如果 OSD 的数量增加并不多，PG 的数量倍增，会不会有什么负收益？

**解答**：
这是一个极具洞察力的问题，触及了 **“何时该分裂”** 以及 **“为什么必须倍增”** 的核心。需要澄清一个关键误区：**“OSD 增加”和“PG 增加”是两码事。**

### 1. 核心误区纠正：OSD 增加 ≠ PG 增加
*   **场景 A：只增加了 OSD（磁盘）**
    *   **操作**：CRUSH 会自动把**现有的 PG（旧箱子）**从塞满的老盘搬一部分到新盘上。这叫 **Rebalance（重均衡）**。
    *   **结论**：这种情况下，**千万不要增加 PG 数量**。你不需要拆分数据，只需要搬运箱子。如果此时强行倍增 PG，确实全是**负收益**，会引发不必要的数据大洗牌，增加系统负担。
*   **场景 B：数据量暴增（单 PG 过大）**
    *   **操作**：当单 PG 数据量 > 100GB 时，OSD 恢复慢、内存容易爆。这时才需要 **Splitting（分裂）**。

### 2. 为什么不倍增（如 4 -> 6）会有灾难？
Ceph 的 PG 计算底层依赖**位掩码（Bit Mask）**。
*   **倍增的秘密（4 -> 8）**：掩码从 2 位变 3 位。只需要检查哈希值的第 3 位。第 3 位是 0 的留旧 PG，是 1 的去新 PG。**只有 50% 数据需要移动**。这是理论上改动最小、迁移最可控的操作。
*   **非倍增的代价（4 -> 6）**：无法使用“多取一位二进制”来解释。Ceph 必须对**所有数据**重新计算一遍哈希取模。**90% 以上的数据归属都会发生变化**。这相当于把书架上的书全部扫下来重新排一遍，引发的 I/O 风暴远超预期。

### 3. 总结与最佳实践
*   **加磁盘时**：永远保持 `pg_num` 不变，让 Ceph 后台自动搬运 PG。
*   **数据量大时**：才考虑倍增 PG。虽然倍增会导致集群短时间内进行大量内部整理，但能从根本上解除“超级大 PG”带来的性能锁死，这是必须付出的代价。
*   **一句话建议**：**加 OSD 不加 PG；只有单 PG 过大时，才按 2 的幂次倍增 PG。**
